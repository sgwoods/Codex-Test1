# Security Issues Resolution Plan

Updated: 2026-06-10

This is the tracked security issue list for Aurora / Platinum release gates. The structured source of truth is `security-issues.json`; this document is regenerated by `npm run security:review:*`.

## Priority Scale

| Priority | Meaning |
| --- | --- |
| P0 | Serious: plausible production compromise, public mutation surface, or integrity break; resolve before beta/production unless explicitly accepted. |
| P1 | Concerning: material exposure or abuse path; reduce before beta when feasible and block production unless acknowledged. |
| P2 | Risky: hardening, minimization, or defense-in-depth gap; track with a concrete owner and release decision. |
| P3 | Warning: hygiene or monitoring issue; keep visible so future code review does not reintroduce it. |

## Release Gate Rule

- `/production` publish must stop when a tracked security issue is open and not explicitly acknowledged in `security-issues.json`, or when an issue marked resolved is still detected in the production artifact.
- `/beta` publish shows the same issues as reminders unless release authority explicitly promotes one to a hard beta blocker.
- Acknowledgement is temporary release-manager acceptance, not resolution. Resolved issues should be fixed in code/assets/config and marked `resolved`.

## Resolution Walkthrough

### P0

#### SEC-002: Production bundle exposes harness and debug mutation APIs

- Status: `open`
- Category: `production-debug-surface`
- Summary: The public production bundle exposes harness/debug globals capable of mutating gameplay state, clock control, scoring paths, replay state, audio capture, and leaderboard test state.
- Latest artifact read: observed (index.html contains window.__galagaHarness__)
- Next action: Add a production build token or lane guard that prevents harness/debug globals from being exported in production and beta unless a review lane explicitly enables them.
- Done definition: Production and beta artifact scans no longer find harness globals or force-write symbols, while local/dev harness checks still pass.
- Evidence:
  - Public bundle includes window.__galagaHarness__ and aliases it to window.__platinumHarness__.
  - Harness methods include stage start, controlled clock advancement, score awarding, forced game over, remote score test setup, local replay seeding, and debug state reads.
  - Production bundle also contains force-write harness flag names.
- Resolution plan:
  - Strip harness/debug APIs from production builds by default.
  - Keep harness exports only in dev/test lanes, or guard them behind a compile-time production false constant.
  - Add production artifact checks for window.__galagaHarness__, window.__platinumHarness__, and force-write harness flag names.
  - Move diagnostic state needed by public support into narrow read-only reports.

#### SEC-003: Leaderboard write integrity depends on client-trusted score submission

- Status: `open`
- Category: `leaderboard-integrity`
- Summary: Score submission is assembled in the browser and sent through the public Supabase client. The anon key is expected to be public, but leaderboard integrity depends on strict RLS and server-side validation that the public artifact does not prove.
- Latest artifact read: observed (index.html contains public Supabase config and index.html contains client score insert)
- Next action: Capture the current Supabase RLS/policy state and classify what the browser can write; then decide whether beta can proceed with a documented leaderboard-integrity limitation.
- Done definition: A release artifact records the audited RLS policy, client writes cannot set trusted verification fields, and production score acceptance has a server-side validation or an explicit accepted-risk note.
- Evidence:
  - Public bundle includes SUPABASE_URL and SUPABASE_ANON_KEY.
  - Public bundle inserts into the scores table from client-side score/game-over state.
  - Read-only passive checks confirmed scores and profiles are publicly selectable with the anon key.
  - No service-role key was observed in the public bundle.
- Resolution plan:
  - Audit Supabase RLS for scores and profiles.
  - Require authenticated users to write only their own user_id and prevent client-controlled verified fields.
  - Move score acceptance to a server-side RPC or edge function that validates a signed run summary or replay evidence where practical.
  - Add a production release check that records the latest Supabase policy audit status before publishing.

### P1

#### SEC-001: Production public artifact exposes reference audio assets

- Status: `open`
- Category: `public-artifact-boundary`
- Summary: The public artifact still exposes reference-audio paths in generated docs, and historical live checks found clip URLs served publicly. Current local production evidence must distinguish public path references from actual bundled private audio bytes.
- Latest artifact read: observed (50 public reference-audio file(s) under assets/reference-audio)
- Next action: Split public guide references from private clip hrefs so production/beta can show evidence labels without publishing or linking private reference-audio paths.
- Done definition: Production and beta artifacts contain no reference-audio clip bytes and no public href/text path that invites fetching private reference clips; dev/local review keeps private provisioning explicit and ignored.
- Evidence:
  - Live production page referenced assets/reference-audio/*.m4a.
  - All 24 referenced public reference-audio paths checked during the passive review returned HTTP 200.
  - GitHub Pages responses included access-control-allow-origin: *.
  - Current local dist/production contains an assets/reference-audio directory but no clip files; generated docs still reference assets/reference-audio paths.
- Resolution plan:
  - Confirm which reference-audio clips are private, licensed, or production-safe.
  - Remove private reference-audio files from production and beta public artifacts.
  - Keep local/dev reference-audio provisioning through ignored local assets or private artifact bootstrap only.
  - Add artifact-boundary checks that fail production if dist/production references or contains private reference-audio paths.

### P2

#### SEC-004: Hosted production lacks explicit browser security headers

- Status: `open`
- Category: `browser-hardening`
- Summary: The live GitHub Pages response did not expose obvious hardening headers such as Content-Security-Policy, frame-ancestors or X-Frame-Options, Referrer-Policy, Permissions-Policy, X-Content-Type-Options, or HSTS.
- Latest artifact read: not observed
- Next action: Run a current hosted-header capture and determine whether a meta CSP is feasible without breaking the inline bundle.
- Done definition: The active host emits documented hardening headers or the app ships the strictest feasible meta policy with explicit residual GitHub Pages limitations.
- Evidence:
  - Passive header check on the live production URL showed GitHub Pages defaults and no obvious app-owned hardening headers.
  - The app is a large inline JS bundle and uses many innerHTML rendering paths, so lack of CSP increases blast radius if any injection bug appears.
- Resolution plan:
  - Decide whether to keep GitHub Pages or front it with a host that can set response headers.
  - Add CSP, frame-ancestors, Referrer-Policy, Permissions-Policy, and X-Content-Type-Options where the host permits.
  - If staying on GitHub Pages, add the strictest feasible meta CSP and document the residual limits.
  - Track any required unsafe-inline exceptions and remove them as build packaging improves.

#### SEC-005: Production build metadata exposes test pilot emails and internal release context

- Status: `open`
- Category: `metadata-minimization`
- Summary: Production build-info and bundle metadata include non-production test pilot emails and detailed commit/release state. This is not a secret leak by itself, but it gives unnecessary phishing and reconnaissance targets.
- Latest artifact read: observed (build-info.json includes platform.auth.nonProductionTestPilotEmails)
- Next action: Sanitize production build-info and production JS token injection so non-production pilot emails and user ids are omitted outside dev/beta review lanes.
- Done definition: Production build-info and production index contain no non-production test pilot emails/user ids, while dev/beta/local auth review still has the needed test-pilot configuration.
- Evidence:
  - build-info.json includes platform.auth.nonProductionTestPilotEmails.
  - The production bundle includes the same configured test pilot emails.
  - build-info.json also exposes commit, branch, promoted-from state, build timing, and release-review context.
- Resolution plan:
  - Remove non-production test pilot emails from production build-info and production JS.
  - Keep only public-safe release metadata in production build-info.
  - Move internal promotion lineage and test account data to private release artifacts.
  - Add metadata minimization checks to production publish preflight.

#### SEC-006: Production bundle mixes dev, beta, and production account/test logic

- Status: `open`
- Category: `release-lane-separation`
- Summary: Production code contains non-production test account rules, harness bypass concepts, and force-write controls. Even when guarded, this expands public attack surface and makes production security reasoning harder.
- Latest artifact read: observed (index.html contains NON_PRODUCTION_LANE)
- Next action: Introduce lane-specific build constants so production strips or hard-disables non-production account/test branches instead of carrying all lane logic in the public bundle.
- Done definition: Production bundle scans no longer find dev/test-only lane symbols unless they are documented as inert public support text; dev/beta behavior remains covered by harnesses.
- Evidence:
  - Production bundle includes TEST_ACCOUNT_EMAILS, NON_PRODUCTION_LANE, testAccountEnabled, and force-write harness flag names.
  - Auth and score behavior branches by release lane inside the public runtime instead of being stripped by build lane.
- Resolution plan:
  - Split lane-specific dev/test controls from production runtime at build time.
  - Keep production account and leaderboard logic minimal and explicit.
  - Add a production bundle scan for dev/test-only symbols.
  - Document any intentionally retained diagnostics as read-only support surfaces.
