# Run Submission Plan

## Goal

Create a low-friction `Submit Run` workflow that can capture:

- the gameplay video generated by the browser
- the matching gameplay JSON log
- a short user note

and turn that into a GitHub issue with permanent artifact links.

## Proposed Direction

Use:

- GitHub issues for tracking and triage
- public Google Drive for artifact storage
- a small upload service to bridge the static game client to Google Drive

## Why This Direction

- GitHub issues are already part of the project workflow
- the video files are too large and awkward for the current email-based feedback flow
- Google Drive is acceptable as low-cost storage if we can make links public and stable enough for review
- the game is hosted as a static site, so browser-only direct integration is not ideal for secrets or upload control

## Recommended Architecture

1. The game records a `.webm` run video and a matching JSON session log.
2. The user clicks `Submit Run`.
3. The game sends both files plus metadata to a lightweight upload endpoint.
4. The upload endpoint stores both files in a dedicated Google Drive folder.
5. The upload endpoint marks the files as `Anyone with the link can view`.
6. The upload endpoint returns public URLs for:
   - the video
   - the JSON log
7. The game or the upload service creates a GitHub issue containing:
   - report type
   - title
   - user note
   - build/version
   - stage/score/lives summary
   - video link
   - JSON link

## Google Drive Feasibility

Yes, but with an important caveat:

- a static GitHub Pages app should not upload directly to Drive using embedded credentials

So if we use Google Drive, the practical implementation is:

- Google Apps Script web app backed by Drive

or:

- a small custom backend using the Google Drive API

## Recommended Google Drive Variant

Use Google Apps Script as the first implementation.

Why:

- low cost
- easy to connect to a specific Drive folder
- can keep Google credentials out of the browser
- can return public share links after upload
- good enough for project-scale sample submission

## Proposed End-to-End Flow

### Browser

- record `.webm`
- generate JSON log
- collect optional note
- POST files and metadata to upload endpoint
- on success:
  - show returned links
  - optionally create a GitHub issue automatically
  - or prefill a GitHub issue URL for review before submit

### Upload Endpoint

- receive multipart upload
- validate size and file type
- write files into Drive folder
- set public-read sharing
- return canonical links
- optionally create GitHub issue using a stored GitHub token

### GitHub Issue Body

Suggested fields:

- Summary
- Report Type
- Build
- Stage
- Score
- Lives
- Browser
- Video URL
- Session JSON URL
- User Notes

## Security And Product Constraints

- Google Drive links will be public if they are meant to be viewable from GitHub issues
- file size limits must be enforced server-side
- abuse controls should exist if the upload endpoint is public
- the browser should not contain Google or GitHub secrets

## Risks

- Google Drive public links are workable but less clean than dedicated object storage
- Drive APIs and sharing settings can be finicky
- upload quotas may become a constraint if usage grows
- GitHub issue auto-creation adds another secret to manage

## Fallback Strategy

If Google Drive proves awkward:

- keep GitHub issues as the reporting surface
- replace Drive with purpose-built object storage later

This keeps the user-facing `Submit Run` concept stable while the storage backend can change.

## Suggested Implementation Phases

### Phase 1

- Add `Submit Run` button
- Upload video + JSON to Google Drive through Apps Script
- Return public links
- Show success UI with copyable links

### Phase 2

- Auto-create GitHub issue from the returned links
- Add issue labels such as `run-report`

### Phase 3

- Add retry, status, and upload-progress UI
- Add artifact retention policy and cleanup guidance

## Decision Recommendation

Proceed with:

- GitHub issues for tracking
- Google Apps Script + public Google Drive folder for storage
- phased implementation so storage upload lands before issue automation
