# Security Issues Resolution Plan

Updated: 2026-06-11

This is the tracked security issue list for Aurora / Platinum release gates. The structured source of truth is `security-issues.json`; this document is regenerated by `npm run security:review:*`.

## Priority Scale

| Priority | Meaning |
| --- | --- |
| P0 | Serious: plausible production compromise, public mutation surface, or integrity break; resolve before beta/production unless explicitly accepted. |
| P1 | Concerning: material exposure or abuse path; reduce before beta when feasible and block production unless acknowledged. |
| P2 | Risky: hardening, minimization, or defense-in-depth gap; track with a concrete owner and release decision. |
| P3 | Warning: hygiene or monitoring issue; keep visible so future code review does not reintroduce it. |

## Release Gate Rule

- `/production` publish must stop when a tracked security issue is open and not explicitly acknowledged in `security-issues.json`, or when an issue marked resolved is still detected in the production artifact.
- `/beta` publish shows the same issues as reminders unless release authority explicitly promotes one to a hard beta blocker.
- Acknowledgement is temporary release-manager acceptance, not resolution. Resolved issues should be fixed in code/assets/config and marked `resolved`.

## Resolution Walkthrough

### P0

#### SEC-002: Production bundle exposes harness and debug mutation APIs

- Status: `resolved`
- Category: `production-debug-surface`
- Summary: Local beta and production artifacts no longer expose browser harness globals, force-write harness symbols, or debug crash globals. Dev still carries the harness for local automation.
- Latest artifact read: not observed
- Next action: Keep the public-lane scan active and run a dev harness smoke check after any future harness/build-lane change.
- Done definition: Production and beta artifact scans no longer find harness globals or force-write symbols, while local/dev harness checks still pass.
- Evidence:
  - Public bundle includes window.__galagaHarness__ and aliases it to window.__platinumHarness__.
  - Harness methods include stage start, controlled clock advancement, score awarding, forced game over, remote score test setup, local replay seeding, and debug state reads.
  - Production bundle also contained force-write harness flag names.
  - Current local beta and production security reviews report SEC-002 not observed after public-lane harness export stripping.
- Resolution plan:
  - Keep browser harness exports only in dev/test lanes.
  - Strip public-lane harness export blocks and debug crash globals during production build and beta promotion.
  - Keep production artifact checks for window.__galagaHarness__, window.__platinumHarness__, and force-write harness flag names.
  - Keep remote write force flags hard-disabled in production even if a public user defines similarly named globals.

#### SEC-003: Leaderboard write integrity depends on client-trusted score submission

- Status: `resolved`
- Category: `leaderboard-integrity`
- Summary: Public beta and production artifacts no longer ship the client-side score insert path. Remote score writes are disabled in public lanes pending server-side validation or a documented RLS policy audit; dev/local harness score-write testing remains available.
- Latest artifact read: not observed
- Next action: Before restoring online score writes in beta/production, capture the Supabase RLS/policy state and implement a server-validated score acceptance path.
- Done definition: Production and beta artifacts do not ship direct client score inserts, or a release artifact records server-side/RLS validation that prevents client-controlled trusted fields.
- Evidence:
  - Public bundle includes SUPABASE_URL and SUPABASE_ANON_KEY.
  - Public bundle inserts into the scores table from client-side score/game-over state.
  - Read-only passive checks confirmed scores and profiles are publicly selectable with the anon key.
  - No service-role key was observed in the public bundle.
  - Current local beta and production security reviews report SEC-003 not observed after public-lane score-submit stripping.
- Resolution plan:
  - Keep public beta/production score writes disabled until score acceptance is server-validated.
  - Audit Supabase RLS for scores and profiles before re-enabling public writes.
  - Require authenticated users to write only their own user_id and prevent client-controlled verified fields.
  - Move score acceptance to a server-side RPC or edge function that validates a signed run summary or replay evidence where practical.

### P2

#### SEC-004: Hosted production lacks explicit browser security headers

- Status: `resolved`
- Category: `browser-hardening`
- Summary: Local beta and production HTML artifacts now include a generated meta Content-Security-Policy and strict-origin-when-cross-origin referrer policy. GitHub Pages HTTP-header hardening remains a future hosting improvement.
- Latest artifact read: not observed
- Next action: After the next hosted beta/dev publish, verify the meta CSP is present in live HTML and capture the remaining host-header limitations.
- Done definition: The active host emits documented hardening headers or the app ships the strictest feasible meta policy with explicit residual GitHub Pages limitations.
- Evidence:
  - Passive header check on the live production URL showed GitHub Pages defaults and no obvious app-owned hardening headers.
  - The app is a large inline JS bundle and uses many innerHTML rendering paths, so lack of CSP increases blast radius if any injection bug appears.
  - Current local beta and production security reviews report SEC-004 not observed after generated meta CSP/referrer policy insertion.
- Resolution plan:
  - Keep the generated meta CSP and referrer policy on beta/production HTML artifacts.
  - Track the residual GitHub Pages limitation that frame-ancestors, HSTS, X-Content-Type-Options, and Permissions-Policy need real host headers.
  - When feasible, front the site with a host that can set app-owned security headers.
  - Reduce unsafe-inline CSP allowances as build packaging moves away from inline bundle/style delivery.
